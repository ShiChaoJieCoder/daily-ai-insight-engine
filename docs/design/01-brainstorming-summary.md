# AI舆情分析日报系统 - 需求分析与设计方案

> 文档创建时间：2026-04-07
> 
> 本文档记录了项目从需求分析到设计方案的完整讨论过程

## 一、项目背景

### 项目目标
构建一个AI舆情分析日报系统（Daily AI Insight Engine），用于：
- 从每日新闻信息中提取结构化洞察
- 生成可读的分析报告
- 提供可视化展示

### 应用场景
- AI行业趋势分析
- 舆情监测与风险预警
- 信息快速理解与决策辅助

### 技术栈要求
- **前端**：React + TypeScript + SCSS + Zustand + Vite
- **设计风格**：参考Impeccable设计理念
- **后端**：Node.js（自选框架）

## 二、核心设计决策

### 2.1 数据获取方式

**最终选择：方案C - 使用RSS/API**

#### 理由
- ✅ 稳定可靠，数据结构清晰
- ✅ 展示技术能力，避免反爬问题
- ✅ 可自动化，适合实际应用

#### 数据源组合：A+B+C
- **技术深度型**：Hacker News API、arXiv API、GitHub Trending
- **行业资讯型**：TechCrunch RSS、The Verge RSS、VentureBeat AI
- **社区讨论型**：Reddit API (r/MachineLearning, r/artificial)、Product Hunt

#### 数据要求
- 至少10-20条AI相关新闻
- 包含：标题、内容/摘要、来源、发布时间
- 中英文混合

### 2.2 数据Schema设计

**最终选择：方案B+C - 深度结构化 + 场景导向**

#### Schema设计（TypeScript类型定义）

```typescript
interface NewsItem {
  // ========== 基础信息 ==========
  id: string;
  title: string;
  summary: string;
  content?: string;
  source: {
    name: string;        // 来源名称：TechCrunch, Hacker News等
    type: string;        // 来源类型：tech_media, community, academic
    url: string;         // 原文链接
  };
  publishDate: Date;
  
  // ========== 分类维度 ==========
  category: 'research' | 'product' | 'funding' | 'policy' | 'discussion';
  tags: string[];        // 自由标签：LLM, Computer Vision, Robotics等
  
  // ========== 实体提取 ==========
  entities: {
    companies: string[];      // 公司：OpenAI, Google, Meta等
    products: string[];       // 产品：GPT-4, Claude, Gemini等
    technologies: string[];   // 技术：Transformer, RAG, Fine-tuning等
    people: string[];         // 人物：Sam Altman, Yann LeCun等
  };
  
  // ========== 事件分析 ==========
  eventType: 'breakthrough' | 'release' | 'funding' | 'regulation' | 'discussion';
  significance: number;  // 重要性评分 1-10
  
  // ========== 分析维度 ==========
  sentiment: {
    score: number;       // -1到1，负面到正面
    label: 'positive' | 'neutral' | 'negative';
    confidence: number;  // 0-1，置信度
  };
  
  impact: {
    level: 'high' | 'medium' | 'low';
    areas: string[];     // 影响领域：industry, research, policy, market
    timeline: 'immediate' | 'short-term' | 'long-term';
    stakeholders: string[];  // 利益相关方
  };
  
  // ========== 趋势标签 ==========
  trendIndicators: string[];  // 趋势指标：scaling_up, open_source, regulation等
}
```

#### 设计理由
1. **基础信息**：满足新闻展示的基本需求
2. **实体提取**：支持关联分析和知识图谱构建
3. **事件分析**：面向舆情监测场景
4. **影响评估**：支持决策辅助和风险预警
5. **趋势标签**：支持趋势判断和热点发现


#### 设计理由
1. **规则处理在前**：先用规则处理能解决的问题，降低AI成本
2. **分层处理**：轻量任务用便宜模型，深度任务用强模型
3. **批处理优化**：减少API调用次数，提高效率
4. **质量保证**：多层验证，确保输出可靠
5. **可维护性**：每层职责清晰，易于调试和优化

### 2.5 AI模型选择与分工

**最终选择：方案C - 按阶段分配任务**

#### 模型分工策略

| 处理阶段 | 使用模型 | 任务内容 | 成本占比 |
|---------|---------|---------|---------|
| **第一阶段：基础提取** | GPT-3.5-turbo / 通义千问 | 分类、标签、实体识别、摘要 | ~40% |
| **第二阶段：深度分析** | GPT-4 / Claude | 情感分析、影响评估、趋势判断 | ~60% |

#### 成本优化策略
1. **批处理**：第一阶段5-10条新闻一起处理
2. **避免重复**：第二阶段只处理结构化数据，不重复处理原文
3. **缓存机制**：相同内容不重复调用API
4. **降级策略**：API失败时使用规则兜底

#### 预估成本（以20条新闻为例）
- **第一阶段**：约 $0.05-0.10（批处理，轻量模型）
- **第二阶段**：约 $0.20-0.30（聚合分析，强模型）
- **总成本**：约 $0.25-0.40 / 每日报告

## 三、设计原则

### 3.1 YAGNI原则（You Aren't Gonna Need It）
- 只实现当前需要的功能
- 避免过度设计
- 保持代码简洁

### 3.2 关注点分离
- 数据获取、处理、展示分离
- 规则处理与AI处理分离
- 前端与后端分离

### 3.3 可维护性优先
- 清晰的目录结构
- 完善的文档体系
- 统一的代码风格

### 3.4 成本意识
- 优先使用规则处理
- 批处理降低API调用
- 缓存避免重复计算

### 3.5 质量保证
- 多层验证机制
- 降级策略兜底
- 人工复核标记
